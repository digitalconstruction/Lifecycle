Level of Detail (LOD)

The acronym LOD has several related expansion - Level Of Detail, Level Of Development, Level Of Definition - that refer to the refinement of information into different directions. The LOD of BIM should be generally asssociated to the key stages of projects when data sharing takes place. This would allow stakeholders to verify that project information is sufficient to meet their requirements, enabling them to decide whether to proceed to the next project stage or not. An ontology schema is shown the figure below. Since different renovation projects can adopt different LOD systems, the developed ontological representation of the proposed LOD framework can accommodate the different representations summarized in the table above.

The LOD frameworks and their levels are represented as classes that can then be instantiated on a project-to-project basis. The class dicl:LODFramework can be instantiated with the frameworks called USA BIMForum, UK LOD, Italian LOD (refer Table 1). Similarly, the levels in different frameworks are added as instances to the class dicl:LODLevel. The link between framework and its respective levels is dicl:hasLevel (inverse: dicl:isLevelOf).

Furthermore, relationships between the levels of a framework are indicated using the object properties dicl:hasNextLevel, dicl:hasSubLevel and their inverse properties dicl:hasPreviousLevel, dicl:hasSuperLevel respectively. A property characteristic called owl:TransitiveProperty is defined for these properties in-order to represent derived relationships between levels. For example, an individual inst:LOD300 has next level inst:LOD400, and inst:LOD400 has next level inst:LOD500, then the ontology can infer that inst:LOD300 has also its next level inst:LOD500.

Another way of defining semantic interpretation of the levels is with the use of axioms. A subproperty of chain axiom (dicl:hasLevel o dicl:hasSubLevel subPropertyOf: dicl:hasLevel) is assigned to the object property dicl:hasLevel. This sanctions the inference that all the sublevels (inst:LOD100, inst:LOD200, etc.) defined within the main levels (inst:AsDesigned or inst:AsBuilt) are also the levels of a specific LOD framework.

The proposed ontology framework is capable to represent the BLCS framework according to different standards, publications, user-defined or project-based specifications.

The adopted methodology is similar to the LOD framework ontology in terms of assigning the stages as a class (dicl:BLCStage). It provides the relationship between the stages using the object properties called dicl:hasSubstage, dicl:hasNextStage and their respective inverse properties dicl:hasSuperStage, dicl:hasPreviousStage. Similarly, the transitive character of the properties is enabled by assigning owl:TransitiveProperty to each object property which further enables the aggregated relationships between the stages. A basic example on the knowledge inference based on the owl:TransitiveProperty is presented in the figure below.

In addition to the property characteristics, some axioms are defined to achieve the complete semantic meaning of BLCS stages into an ontological framework, shown in figures above. In an example scenario of BS EN 16310 framework represented in the figure below, each stage is comprised of many sub-stages and technically these sub-stages are also stages to that framework. This semantic meaning is inferred in ontology with the help of the subProperty chain axiom (dicl:hasStage o dicl:hasSubStage subPropertyOf: dicl:hasStage) defined to the object property dicl:hasStage. Similarly, in the same scenario, the main stage inst:Initiative has a next stage relation with the substage called inst:ProjectInitiation. These complex relations are modelled in the ontology by using axioms dicl:hasNextStage o dicl:hasSubStage subPropertyOf: dicl:hasNextStage, dicl:hasPreviousStage o dicl:hasSubStage subPropertyOf: dicl:hasPreviousStage assigned to properties dicl:hasNextStage and dicl:hasPreviousStage respectively.

Property state representation

The figure below shows the relationships between Activities, BIM object data and its LOD for a specific activity in the project. A difficult representational challenge in RDF is how to represent the values of the same properties of objects at multiple different levels. The most typical approaches are to use objectified properties or named graphs of RDF datasets. In the following, the former approach objectified properties is adopted. Along with the Digital Construction ontologies, existing work on Building Topology Ontology (BOT) in the Linked Building Data (LBD) community and ifcOWL ontology is used. In the Lifecycle ontology, a class dice:BuildingObject is produced to hold the information of the objects and is aligned to classes of existing ontologies (bot:Element, bot:Space and ifcowl:IfcObject) using rdfs:subClassOf property. This dice:BuildingObject is further interrelated to activities (dice:Activity) in the project using the object property dicl:requires/tt> and its inverse property dicl:wasNeededFor.

The classes dicv:Property and dicv:PropertyState in Digital Construction Variables ontology make it possible to add an ultimate number of properties to building objects and their growth of accuracy throughout the project life-cycle. Additionally, it supports for defining meta-data attributes for each property (e.g., role or quantity kind, unit) and each property state (e.g. source, timestamp, value, etc.). Fig. 9 indicates these capabilities for instances of OPM classes. The developed ontology also supports the modelling of relationships between LOD and BIM attributes by using the object property dicl:hadLODLevel and its respective inverse property dicl:isLODLevelOf.

The next phase of the ontology development is to enable modelling of the data flow between Agents (either Stakeholder or Actor in a project) according to their requirements and involvement in the activities specified by the BLCS. In the Lifecycle Ontology, the class dice:Agent specifies all actors/stakeholders involved in the building renovation process. Actors and stakeholders are instances of the class dice:Agent. A role is assigned to them based on their involvement in the specific activity (dice:Activity). A class dicl:Role accommodates different roles (dicl:InformationConsumer, dicl: InformationProcessor, dicl: InformationProvider) as its subclasses and is assigned to the agent through the object property dice:hasRole and its inverse property dice:isRoleOf.

A simple relationship is defined between activity and the agent by using the object property dice:Agent and its respective inverse property dice:isAgentIn. Furthermore, the object property dice:isAgentIn is categorized into three sub-properties dicl:consumesFrom, dicl:providesTo and dicl:processFrom, which are used based on the role of the agent. In common practice, the activities are listed according to the BLCS of the project and the same is represented in the ontology using the object property called dice:hasState and its inverse property dice:hasActivity. The class dice:Activity is also directly related to the BIM object data.
